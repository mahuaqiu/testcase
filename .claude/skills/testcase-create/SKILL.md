# 自动化测试用例生成

直接执行测试用例生成全流程，无需Agent调用。需要AW操作时调用 testcase-aw Skill。

---

## ⚠️ 强制规则（违反视为执行失败）

1. **禁止跳过用户确认**：所有确认点必须使用 `AskUserQuestion` 工具
2. **禁止提前生成代码**：步骤6前禁止生成任何测试代码或AW代码
3. **AW操作强制调用 testcase-aw**：发现需新建AW立即停止，调用 `Skill(skill="testcase-aw")`
4. **流程不可压缩**：步骤1-7必须完整执行，每个步骤输出必须符合规范格式

---

## 执行流程

```
步骤1: 读取项目规范
    ↓
步骤2: 分析用户输入（步骤完整性校验）
    ↓
步骤3: 查找相似用例（优先复用）
    ↓
步骤4: AW匹配分析 → ⚠️ 发现需新建AW → 立即停止 → 调用 testcase-aw
    ↓
步骤5: 输出计划 → ⚠️ AskUserQuestion 等待用户确认
    ↓
步骤6: 生成测试代码（仅用户确认后执行）
    ↓
步骤7: 复查验证
```

---

### 步骤 1：读取项目规范

**必须**先读取：`AGENTS.md`（架构规范）、`aw/INDEX.md`（AW索引）、`config.yaml`（hooks配置）

---

### 步骤 2：分析用户输入

#### 2.1 信息提取

| 检查项 | 必要性 |
|--------|--------|
| 测试端 | 必要 |
| 测试步骤（STEP 1, STEP 2...） | 必要 |
| 期望结果 | 必要 |
| 前置步骤/角色/清理步骤 | 可选 |

#### 2.2 步骤完整性校验（强制）

**必须**输出校验表格：

```
| 用户STEP | 解析内容 | 覆盖状态 |
|----------|----------|----------|
| STEP 1 | 登录操作 + 断言成功 | ✅ |
| STEP 2 | 创建会议 | ✅ |

校验结论: ✅ 全部覆盖 / ❌ 有遗漏需补充
```

#### 2.3 角色命名自动转换

用户输入角色 → 标准命名：主持人/A → `userA`，与会者/B → `userB`，嘉宾C → `userC`

#### 2.4 操作类型自动识别

"api预约会议" → API，其他UI描述 → UI。不明确时才询问。

---

### 步骤 3：查找相似用例（优先复用）

**在查INDEX.md之前**，先在目标目录查找相似用例。

#### 3.1 查找策略

优先查目标目录（最多2个），不足时查隔壁目录（最多2个），总数不超过4个。

#### 3.2 输出复用分析

```
| 用户STEP | 可复用AW方法 | 复用状态 |
|----------|--------------|----------|
| STEP 1 | LoginAW.do_login | ✅ 可复用 |
```

#### 3.3 复用决策

完全匹配 → 直接复用；部分匹配 → 复用匹配部分，补充缺失；无匹配 → 正常AW匹配流程。

---

### 步骤 4：AW匹配分析

#### 4.1 匹配优先级

| 匹配类型 | 处理方式 |
|----------|----------|
| 完全匹配 | 直接使用已有AW方法 |
| 部分匹配/无匹配 | 调用 testcase-aw |

#### 4.2 输出匹配表格

```
| 用户STEP | AW类 | 方法 | 匹配状态 |
|----------|------|------|----------|
| STEP 1 | LoginAW | do_login | ✅ 已有 |
| STEP 3 | - | - | ⚠️ 需新建 |
```

#### 4.3 需要AW操作时（⚠️ 强制检查点）

发现需新建/扩展AW时：**立即停止** → 调用 `Skill(skill="testcase-aw")` → 等待确认 → 继续步骤5

**禁止**：直接生成AW代码、跳过 testcase-aw、假设用户同意

---

### 步骤 5：输出计划，用户评审（⚠️ 强制确认点）

**⚠️ 用户确认前禁止**：生成代码、创建文件、假设用户同意

**只允许输出**：分析表格、计划文档、AskUserQuestion调用

#### 5.1 计划输出格式

```markdown
# 测试用例生成计划

## 基本信息
- 测试文件: testcases/{平台}/{模块}/test_{功能}_{场景}_{编号}.py
- 用户资源: {"userA": "{平台}", "userB": "{平台}"}

## 步骤与AW映射
| 用户STEP | AW方法 | 操作类型 | 状态 |
|----------|--------|----------|------|

## 需要新增的AW（如有）
| AW类 | 方法 | 平台 |
|------|------|------|
```

#### 5.2 用户确认（必须 AskUserQuestion）

调用 `AskUserQuestion`：
- 问题："请评审以上生成计划，确认是否执行？"
- 选项：确认执行 / 需要修改 / 取消

**用户选择"确认执行"后才进入步骤6**。

---

### 步骤 6：生成测试代码

用户确认后执行代码生成。

#### 6.1 测试用例模板（核心结构）

```python
"""{用例标题}测试用例。"""
import pytest
from common.parallel import parallel

@pytest.mark.users({"userA": "{平台}", "userB": "{平台}"})
class TestClass:
    def test_{功能}_{场景}_{编号}(self, users):
        userA = users["userA"]
        userB = users["userB"]

        # 步骤 1: 并行登录
        with parallel():
            userA.do_login()
            userB.do_login()

        # 步骤 2-N: 按计划执行
        # ...
```

#### 6.2 模板要点

- 导入顺序：pytest → 项目模块
- API用户：自动创建 `_api` 实例
- 并行操作：`with parallel():` 包裹
- 步骤注释：每步标注编号

#### 6.3 更新INDEX.md

新增AW后，更新 `aw/INDEX.md`（格式见AGENTS.md）。

---

### 步骤 7：复查验证

#### 7.1 步骤覆盖验证

对比用户STEP与代码实现，遗漏时自动补充。

#### 7.2 hooks冲突验证

检查清理操作是否与 `config.yaml` 的 `hooks.teardown` 重复，重复则移除。

#### 7.3 编码规范验证

检查：文件命名、类命名、方法命名、AW引用路径、Docstring（中文）

#### 7.4 INDEX.md同步验证

检查新增AW是否已添加到INDEX.md。

#### 7.5 输出复查结果

```
✅ 复查通过 或 ⚠️ 发现问题已修正：
- 步骤覆盖：{N}/{N}
- hooks冲突：无/已移除重复项
- 编码规范：符合/已修正
- INDEX.md：已同步/已补充
```

---

## 推荐原则

1. **优先复用相似用例**：查INDEX.md前先查同目录用例
2. **生成后复查**：步骤覆盖、编码规范、INDEX.md同步
3. **遵循命名规范**：详见AGENTS.md

---

## 命名规范速查

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| AW文件 | {业务名}_aw.py | login_aw.py |
| AW类 | {业务名}AW | LoginAW |
| AW方法 | do_{动作} / should_{期望} | do_login() |
| 测试文件 | test_{功能}_{场景}_{编号}.py | test_login_success_001.py |
